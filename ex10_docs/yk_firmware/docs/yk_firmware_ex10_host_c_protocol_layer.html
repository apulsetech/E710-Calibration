Ex10Protocol Layer¶

The Ex10Protocol interface provides an abstraction of the wireline protocol. The Ex10Ops object uses this interface to abstract the sequence of register writes/reads necessary to implement the various operations. The Ex10Protocol interface uses the Ex10CommandTransactor and Ex10Command modules listed below to create the wireline transactions.

Ex10Commands¶

The Impinj reader chip commands layer provides the C language functional interface to the Impinj reader chip over the wireline communication interface.

The commands layer communicates to the Impinj Reader Chip using the Ex10CommandTransactor interface.

The commands layer does not provide thread safety. Thread safety is maintained when the commands layer functionality is accessed from the Ex10Protocol Layer.

Buffer allocation¶

The command and response buffers are statically allocated within the commands layer commands.c source file. These buffers are allocated using the symbol EX10_SPI_BURST_SIZE, which is defined within the board layer board_spec_constants.h header file.

The command layer accepts arbitrary length operations via the Ex10Commands interface. These operations are divided up into EX10_SPI_BURST_SIZE sized commands which are transferred across the wireline interface to/from the Impinj Reader Chip. Specifically the Read, Write, and ReadFifo commands are divided into transfer sized chunks in this manner.

Note: The WriteInfoPage command does not provide this capability. This specific command requires that data be transferred in a single host interface (SPI) transaction. Therefore this command cannot be divided into smaller sections for transfer. The Ex10Commands.write_info_page() interface implementation may need to be modified by the OEM to suit their needs. The reference design implementation allocates the transfer buffer size to EX10_BOOTLOADER_MAX_COMMAND_SIZE; which is 2048 + 4 bytes.

struct Ex10Result (*read_fifo)(enum FifoSelection selection, struct ByteSpan *byte_span)¶

Read the specified number of bytes from device FIFO stream.

Given a selected fifo, read bytes from the fifo into a ByteSpan.

Multiple ReadFifo commands are made until the byte_span length value is filled to capacity.

The byte_span->length should be the number of bytes obtained by reading the EventFifoNumBytes register. It does not include the response code byte.

The byte_span->length value will be updated with the number of bytes read using the ReadFifo command into the buffer.

Note

If 32-bit alignment of the event fifo messages is required then byte_span->data[1] must be u32 aligned since data[0] will contain the response code.

Warning

If the byte_span value is greater than the number of bytes contained in the fifo then this function will continue to read from the fifo until the length value is satisfied. Therefore the byte_span length value should contain the value read from EventFifoNumBytes register.

Param selection

Which FIFO to read from. In the Ex10 only the EventFifo (0) is supported.

Param byte_span

A struct with a pointer to a ByteSpan struct to hold the requested bytes, along with a length. The byte_span data buffer should be able to hold one more byte than the number of bytes to read from the fifo. This extra byte will hold the response code from the device.

Return

Returns the Ex10Result struct to inform the caller what type of issues happened during the command. Can return a host_result error if the passed data address for reading the fifo is not 32 bit aligned. Can return a device_response error if the first byte read back from the device is not a Success code. Can return a host_result error if the length read back is not what was expected from the response.
