Ex10 Ramp Module Manager¶

The Ex10RfPower module supports pre- and post-rampup callbacks. The Ex10 Ramp Module Manager is a block that is used to manage those rampup callbacks for the user. The purpose of this manager is to provide a mechanism to disconnect the pre- and post-rampup callback features from the other parts of the SDK, this way they can be used with other components of the SDK independently without modification. This allows these features to be added to any of the use cases for any given example. The module manager constrains that the pre and post callbacks are defined as a set to ensure that the compatibility and performance of the callbacks have been verified.

Some of the rampup callback modules require information about how the Impinj Reader Chip is configured. The way that this module manager decouples the rest of the SDK from these needs is that any module that calls cw_on() in the Ex10RfPower module must always set the pre- and post-rampup variables by calling the store_pre_ramp_variables() and store_post_ramp_variables() with the appropriate values. This way the use case just provides the information that the modules will need regardless. These store_... functions take a list of all of the parameters that might be needed by any of the modules. These functions were purposefully written to collect all of the variables possibly needed by a module in a single call. If this list changes in the future because a new module has been added and needs a new parameter, this will cause the SDK/applications to fail to compile and can be addressed when the change is made. These parameters are stored in the ex10 ramp module manager, so that they can be retrieved by the modules if needed. Any block can use the ex10 ramp module manager as a storage mechanism (it is not restricted to the ramp modules).

The module manager also stores the function pointers that will be called before and after the Ex10RfPower module ramps up. This way the Ex10RfPower module is decoupled from managing the function pointers and can just call the call_pre_ramp_callback() and call_post_ramp_callback() functions at the appropriate times. If the rampup callback functions return an error, then the rampup will be aborted. The register_ramp_callbacks() function registers both the pre- and post-rampup callbacks at the same time and will return an error if there are any callbacks already registered. It does this to couple the pre- and post- behavior together so that if a module needs to have both a pre and post callback, another module does not break that module by re-registering just one of the rampup callbacks on its own.

The ex10 ramp module manager only allows one module to be installed at a time. This is so that Impinj can verify that the modules operate as expected. If there are situations where multiple modules are useful together, then a super module will be created and Impinj will verify that the operation of the combined modules is optimized and handle any possible conflicts. Of course the user can create their own modules, but they are then responsible for verifying the modules’ proper operation.

The Ex10 ramp modules must implement a basic API that is an init() that will register their pre and post callbacks with the Ex10 ramp module manager, and a deinit() that will remove the callbacks. The modules can also have other API calls that can be specific to that module. This init() API should be called from the application layer which knows that the modules features are desired. This way the pre- and post-ramp modules can be mixed and matched with the use cases at the application level.

struct Ex10RampModuleManager¶

This is the pre and post ramp callback module manager.

It provides an interface where the modules can register themselves for callback, and the use cases can call a standard function that will then determine if any callback is registered.

This module also is a place where state variables can be stored for the callbacks to use. The use cases are responsible for maintaining these variables to be accurate when the callbacks are called. This allows the callback modules and the use cases to be decoupled from each other.

Public Members

void (*store_pre_ramp_variables)(uint8_t antenna)¶

These functions are used to capture the values needed for the pre and post ramp callbacks.

It is the responsibility of the use case/reader layers to set these before the callbacks are called.

These are the current settings needed by the supported modules and the list of parameters may grow in the future. If so, these functions should grow to add those new parameters. This is because the use cases that are setting these variables don’t know what modules are enabled, so they should store all of the parameter. And if a new parameter is added for a new module, it will change the function signature and cause the use cases to fail to compile until they add the new parameter.

Registers the function pointer the rf power module to call before and after ramping.

Param pre_cb

The function pointer to register. The function takes a pointer to Ex10Result as a parameter. In the case of an error, any error handling should be done by the callback context since it knows what is appropriate to do for the error. If the Ex10Result.error is set to true, it will cause the cw_on to abort the rampup and return Ex10Result.

Param post_cb

The function pointer to register. The function takes a pointer to Ex10Result as a parameter. In the case of an error, any error handling should be done by the callback context since it knows what is appropriate to do for the error. If the Ex10Result.error is set to true, it will cause the cw_on to abort the rampup and return Ex10Result.

Return

Returns Ex10Result with success if successful or an SDK error if there are already a callbacks installed.

Antenna Disconnect¶

The Antenna Disconnect module has only post-rampup functionality. The module has default values that are setup for the E710 Development Board. Some of these parameters are defined in the board_spec.h file and some of them have default values. The API allows these default values to be modified if desired.

Listen Before Talk¶

The Listen Before Talk module is used to verify that there is no other reader already occupying the channel that the Impinj Reader Chip is planning to ramp up on. It does this by measuring the RSSI at the RX pin and will not return until it detects that the channel is unoccupied. It also has a timeout feature that will return a fail code if the channel was never clear.

The parameters for the LBT operation have been pre-configured for the Japan 2 region and can be modified by using the setters in the API.

Runs the LBT op and retrieves a single RSSI measurement through the measured_rssi_log2_reg register.

This value is then thrown through the calibration layer compensation before being returned to the user.

Param antenna

The antenna to listen on.

Param frequency_khz

If a frequency is specified here, this forces the stack to always ramp up to the same frequency. If 0, the region specific jump table will be used.

Param lbt_offset

The offset frequency for measuring the RSSI.

Param rssi_count

The integration count for measuring RSSI.

Param override_used

Determines whether or not to use the Rx gain override setting in the LBT control register. This is normally false, which means the default LBT-specific settings are used in RSSI measurement and compensation. These default settings override the RxGainControlFields in the RxGainControl register. If the override is set, the user-specified settings in the RxGainControl register will be used during measurement and compensation. Note that these user settings must be set before this function call.

Param lbt_rx_gains

The rx gains to use during the lbt measurements.

Param lbt_rssi_out

The compensated RSSI value in cdBm.

Return

The status showing any errors which occurred.

Runs the LBT op with a variable number of measurements.

The user configures the number of measurements up to 5 along with the delay between each measurement. This delay will be the same between all measurements. The user may also set the LBT offset and base frequency to a different value for each measurement.

Note

: The override field determines whether or not to use the Rx gain override setting in the LBT control register. This is normally false, which means the default LBT-specific settings are used in RSSI measurement and compensation. These default settings override the RxGainControlFields in the RxGainControl register. If the override is set, the user-specified settings in the RxGainControl register will be used during measurement and compensation. Note that these user settings must be set before this function call.

Param antenna

The antenna to listen on.

Param frequency_khz

If a frequency is specified here, this forces the stack to always ramp up to the same frequency. If 0, the frequency used will iterate through the region specific jump table.

Param rssi_count

The integration count for measuring RSSI.

Param lbt_settings

The lbt controls for the entire op. This tells the op whether to use a delay between measurements, how many measurements to make, whether or not to override the RX gain settings, and if a narrower bandwidth should be used.

Param frequencies_khz

The frequencies array for all LBT measurements. Note that these can all differ.

Param lbt_offset

The offset frequency for measuring the RSSI in each measurement. Note that these can all differ.

Param rssi_measurements

The rssi results are placed in this array. Ensure that the array contains enough space for the number of measurements that were specified in lbt_settings.

Param lbt_rx_gains

The rx gains to use during the lbt measurements.

Return

The status showing any errors which occurred.

Antenna Disconnect and Listen Before Talk¶

The antenna disconnect and listen before talk is a super module that combines the functionality of the Antenna Disconnect module and the Listen Before Talk module together so that they can be used at the same time. This module has taken into account interactions between the modules. Note that this function actually uses the ramping callbacks from the two modules directly as they do not interact with each other.

The parameters for the antenna disconnect and the listen before talk module should be set using the setters on this API so that it can take into account any interactions that the modules might have.
