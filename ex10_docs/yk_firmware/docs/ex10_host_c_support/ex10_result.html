Ex10 SDK Error Reporting¶

The Ex10 SDK integrates a single path for error reporting. This enables a streamlined path to handle and troubleshoot errors that occur at any point in an Ex10 based reader, including errors that occur in the firmware and in the C SDK. At the core of error reporting scheme is the Ex10Result structure, as described below.

As soon as a failure happens, a struct Ex10Result is created to capture the error specifics at the earliest point. It is to be passed to a higher level for analysis and handling. Once a caller identifies that struct Ex10Result contains an error condition, it will be passed upwards without modification.

On success, error field will be set to 0 and other fields should be ignored.

On error, error field will be set to 1 and other fields will be set with relevant information:

  • the Ex10Module where the error originated

  • result code identifying the error type (Ex10SdkResultCode or Ex10DeviceResultCode)

  • Additional error-specific information will be provided in device_status field

The struct Ex10Result can be extended to report customer-specific error conditions. When customer bit is set to 0, the Ex10Result struct format is as shown in the following table. When customer bit is set to 1, a different internal structure can be defined, with the following 2 limitations:

  • Overall Ex10Result size cannot exceed 64 bits

  • error bit logic remains consistent, meaning 0 denote success and 1 denote that an error occurred

Interrupt Context Errors¶

Errors may occur during interrupt processing and these require special handling. In this case the Ex10 SDK interrupt context cannot “return an error to caller” because it was triggered by an interrupt from Impinj Reader Chip and not called by the application.

Currently the interrupt context uses the Event FIFO queue to communicate with the main context, so same approach will be used to report to the main context on errors occurred in interrupt context.

When an error occurs in interrupt context, a Ex10Result Event FIFO packet will be created and enqueued into Ex10 Event FIFO queue to be processed by the application. This new packet will contain a struct Ex10Result with specific details on the encountered error, as done in main context error reporting.

Interrupt context has a buffer pool dedicated for error reporting. If the interrupt context encountered an error, but the error pool is empty, that error will be silently discarded.
