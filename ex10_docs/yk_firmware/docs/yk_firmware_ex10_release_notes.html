Impinj Reader Chip SDK Release Notes¶

This bundle contains the Impinj reader chip SDK and the firmware images for 2.00.00.

Features added in 2.00.00¶

  • Revamped the LBT op to allow running multiple measurements

    • The LBT settings register now allows control of number of measurements as well as a delay time between each. The number of measurements also caps out at 5. The LBT op usage in the C-SDK has also changed to utilize this new functionality and improve the total LBT time (dependent on environment).

  • Support for Impinj proprietary MarginRead command

    • Margin Read allows the user to check if a tag is working as expected by way of checking the data bits in non-volatile memory. Further documentation can be found on the Impinj portal.

  • Impinj reader chip SDK build system now uses CMake

    • This allows for a more maintainable and customizable build system. See Building the C SDK sources.

  • Revamp usage of regulatory regions

    • The active region is now a separate module and all regions can be accessed separate from the active region. Regions are now customizable and can be changed on the fly. There is a new functionality to set up regions with a single frequency and update the channel list. For details, see Regions Introduction.

  • Unified error returning across Impinj reader chip SDK

    • struct Ex10Result was created as a common error structure throughout the Impinj reader chip SDK.

      • struct Ex10CommandsHostErrors replaced by struct Ex10Result

      • struct OpCompletionStatus replaced by struct Ex10Result

      For details, see Ex10 SDK Error Reporting.

      As part of this effort, assert statements were removed from core SDK sources.

  • MemoryDiagnosticMode support (EnhancedIntegra)

    • Added a MemoryDiagnosticMode field in a new TagFeaturesControl register to enable reporting of memory parity self-check errors as supported by Impinj M700 series endpoint ICs.

      If MemoryDiagnosticMode is enabled and the endpoint IC has detected a parity error, memory_parity_err will be set in the tag_read event FIFO packet.

  • New RF modes

    • RF Mode 226

      This mode is targeted for the Japan region with read rates of 500 tags/s, with LBT op. The mode parameters are M2/426, 12.5us Tari, 1.5:1 PIE.

    • RF modes 187, 287 and 384

      Highly sensitive modes, intended to be used as part of Autoset feature. All 3 modes are functionally identical, sharing the same link parameters and performance.

    • RF mode 203

      This mode is targeted for the Japan region with read rates of 600 tags/s, with LBT op. The mode parameters are M0/426, 12.5us Tari, 1.5:1 PIE.

    • RF mode 225

      This mode is targeted for the ETSI LB region with read rates of 400 tags/s. The mode parameters are M2/426, 15us Tari, 2:1 PIE.

  • Ease of use improvements for Impinj reader chip SDK

    • Removed dynamic size arrays from the SDK

      • All variable sized arrays have been removed from the SDK. This provides for a more deterministic stack size and compiler compatibility.

    • printf and fprintf statements removed, replaced by board-specific API

    • Improved printed messages consistency for SDK sources and examples.

      • As part of this effort, ex10_printf() and ex10_eprintf() are used in SDK sources to print messages are error respectively. The examples similarly use ex10_ex_printf() and ex10_ex_eprintf().

    • Improved structures initialization and clean up of additional compiler warnings

    • Abstract OS-specific system calls for improved portability

    • Better organization for core SDK modules

      • For details on updated SDK organization, see C Host SDK API Layers

    • New Use cases layer, for details see Ex10 Use Case Layers

  • Autoset example and support for inventory sequencing in SDK

    • Added an Autoset example that uses sequence of inventories to find the most number of tags in the field of view in the least amount of time.

  • CalGen tool created for calibration customizations

    • For details, see Calibration Customization Using CalGen

  • Added FIFO overflow packet

    • Added new FIFO overflow packet for when the FW attempts to add a FIFO packet but there is not enough space. More info on the packet can be found in the packet description FIFO Overflow

  • FCT wrapper example can now do arbitrary mux routing

    • The ‘y’ command in the FCT wrapper example now allows setting values to each of the four signal multiplexers instead of just routing the PGA3 to ATEST.

  • A new module ex10_rx_baseband_filter.h is added to the board layer

    • The module allows to select the Dense Reader Mode (DRM) status and choose the RX baseband filter type based on the DRM status and the current RF mode.

Fixes in 2.00.00¶

  • Calibration flash page can be easily/accidentally locked (PI-28323)

    • Avoid unintentional cal info page locking by defaulting to 0xFFs in the lock field of the calibration page and adding an “allow lock” param to pc_cal_example.py.

  • Enabling TagFocus disables other Select commands (PI-32416)

    • Enabling TagFocus feature interfered with configured Select commands. When TagFocus was enabled, the SEL field of Query was forced to ‘1’, allowing all existing tags in the field of view to reply regardless of their SL flag, and ignoring the configured Select commands.

      Although there is no restriction in using any field configurations for InventoryRoundControl, when enabling TagFocus, Impinj recommends running single target inventory (either Target A or Target B) in Session S1. Note that TagFocus only has an impact when inventorying in Session S1.

  • Typical FCC occupancy may exceed 400 ms per channel (PI-33509)

    • Nominal channel timer in FCC region is set to 200 ms. Due to FW/HW latency to turn the transmitter off, the actual dwell time on a channel in FCC region was 206 ms, thus exceeding the allotted limit set by FCC of 400 ms per channel in 20 s window.

      To assist with this issue, a board_regulatory_overshoot_ms variable was added in Ex10 regulatory module. This value can be adjusted per-design and it is subtracted from the nominal timer to account for this overshoot.

  • Continuous inventory in ex10_reader masking some inventory op errors (YF-2147)

    • Continuous inventory implemented in fifo_data_handler() of Ex10 Reader was continuing inventory when previous inventory op reported an error.

      New behavior will stop continuous inventory when an error is reported via the stop reason field in Inventory round summary Event FIFO packet and report the error to the caller via the Continuous Inventory summary Event FIFO packet

  • Protected mode example uses wrong PIN byte order (PI-32856)

    • Improper byte ordering for the Access password (PIN) in the Protected Mode example.

  • ETSI Burst example used different timing parameters from ETSI Spec​ (YF-2025)

    • ETSI burst example used wrong durations (on timer and off timer) for the ETSI spec.

  • Return Loss threshold detection improvements (PI-31174​, PI-32035​)

    • return_loss_cdb = 0 resulted in aborting due to return loss, for any amount of return loss present at the antenna port.

    • When return loss threshold level was beyond the LO cal level, return loss detection mechanism showed unexpected behavior.

  • pc_cal_example.py ETSI low band calibration (PI-33599)

    • When the SKIP_LOWER_BAND_FREQ_CAL option is enabled in pc_cal_example.py the power shifts for the skipped frequencies are now set to zero instead of doing calculations with invalid values.

  • pc_cal_test.py script fixed to run from bundle without modifications (PI-29865)

    • When running pc_cal_test.py from an extracted v1.2 release bundle, imports with cal. prefix fail. To fix, cal. prefix was removed from the imports, as was done in pc_cal_example.py.

  • Multiple Select commands not working properly with RF mode 244 (PI-30623)

    • With RF Mode 244 (FCC DRM), sending multiple arbitrary Select commands with a single call to SendSelect op would break the T4 timing limitation in the UHF Gen2 spec​. To fix, the Impinj reader chip FW was updated to observe proper T4 delay between consecutive Select commands.

  • event_fifo_buffer_peek() race condition and logic bug (PI-30744)

    • Retrieval of list_front(..) was not guarded, which may lead to data corruption due to a race condition between reading and inserting to reader’s Event FIFO list. To ensure proper operation, this access is now guarded by reader list mutex, similar to reader_list_node_push_back(..).

      In addition, a check of list_node after retrieving list_front(..) was removed. When the list is empty, a pointer to the sentinel node is returned. The sentinel node member data will be NULL, indicating that the pointer to the list node is a sentinel node. Therefore, the check if (list_node) { .. } will always be true.

  • RSSI error with multiple antennas due to calibration antenna LUT misinterpretation (PI-32658)

    • The RSSI compensation for the antennas used the antenna number field rather than the antenna offset field. get_gain_offset(..) function was fixed to use the correct parameter.

  • Ex10Protocol.reset() bugs fixed (PI-30295)

    • The incorrect assignment operation was used (error = 0) instead of the test for equality operator (error == 0). This causes the conditional to fail. Since the condition fails, the reset SPI clock rate is always set to Bootloader speed and never upgraded to the faster Application speed.

    • If the requested destination is Application, but the ‘running location’ is Bootloader, then the correct error code of 1 is not returned. Returning ‘1’ indicates that destination result of Application was not satisfied.

  • Continuous inventory round done fixed (PI-26990)

    • The continuous inventory stop condition for the number of rounds done would count the number of inventory round ops that were completed, not the number of inventory rounds that were completed. The fix is to only count the inventory round as done when the inventory summary reason is Done or Host.

  • The aggregate op will now check for errors in executed EX10 commands (PI-27961)

  • Negative Tx power error when droop compensation is enabled and transmitter is unable to reach target power(YF-2062)

    • When operating for a long time without hopping, droop compensation may adjust Tx scalar to a value outside of the valid operating range, causing negative Tx power error.

      This issue was resolved by limiting the Tx scalar values set by droop compensation mechanism to the operating range.

Known issues in 2.00.00¶

  • Connecting Aux DAC to ATEST2/3 lowers TARI (YF-2123)

    • The produced TARI value changes when using the ATEST mux to connect the AUX DACs to ATEST2 and ATEST3 (value is reduced by ~10%).

      A suggested workaround is to connect another signal to ATEST0/1. For example:

      set_atest_mux(1u << 15u,1u << 15u,1<<19,1<<19); //connect both CTfilter out and DACs to ATEST

      set_atest_mux(1u << 14u,1u << 14u, 1<<19, 1<<19); //connect both AuxADC inputs and DACs to ATEST

  • TxCoarseGain register can be set to invalid value of 31 (YF-2097)

    • Impinj reader chip firmware does not produce an error on attempt to set Tx attenuation to invalid value of 31. As stated in TxCoarseGain register documentation, valid values are integers in [0, 30] range and future firmware version will be updated to generate an appropriate error for all values outside of that range. It is not recommended to set Tx attenuation to a value outside of the supported range.

  • Uncalibrated/default calibration behavior may select invalid Tx coarse attenuation (YF-2098)

    • When using an uncalibrated device and commanding to Tx power outside the recommended range, the produced Tx coarse gain values may be outside of the supported range of [0, 30].

      Note that uncalibrated devices may be used for engineering/development purposes only and should not be deployed to production. Please refer to the calibration app note for additional details on using an uncalibrated device.

  • Impinj reader chip SDK Indonesia region may include an incorrect channel ID (YF-2099)​

    • Channel table for Indonesia may incorrectly use channel 30 instead of 40. For impacted application, the usable channel list for Indonesia region may be manually updated in src/ex10_regulatory/ex10_regulatory_regions/ex10_regulatory_indonesia.c.

  • E Family-based readers may struggle to pass ETSI lower band spectral mask at transmit powers >31.5 dBm (YF-2153)

    • The Inventory and ETSI Burst sequences implemented in the E Family SDK and firmware produce spurious emissions that may make it difficult to pass the ETSI LB spectral mask as defined in ETSI EN 302 208 at transmit power targets above 31.5 dBm.

    • Impinj is actively working on improving spectral performance. Please reach out to your account executive for more details.

  • Gen2 Untraceable command is not supported (YF-2169)

    • Gen2 Untraceable command is not supported. Attempts to execute a Gen2 Untraceable will be rejected and a Gen2 Transaction Event FIFO packet with status unsupported will be issued.

  • Tag user memory Reads are limited to maximum of 61 words (122 bytes) (YF-986)

    • Maximum supported number of words that may be read from a tag using Gen2

    Read command is 61 words.

  • ADC measurements give bad results if they coincide with Tx power droop compensation (YF-2051)

    • Tx power droop compensation should be disabled when making ADC measurements to ensure accurate ADC measurement results.

  • Using an unlimited regulatory timer may cause poor reader performance (PI-29431)

    • The Impinj Reader Chip SDK performs certain reader chip configuration steps only when the reader is ramped down and ramped back up. For example, temperature measurement for LO PDET ADC target selection and PLL configuration. If the reader temperature changes drastically without the reader ramping down and back up, the TX power will become inaccurate, and the PLL could become unlocked. Impinj recommends partners do not use an unlimited regulatory timer for this reason. Impinj’s implementation of the regions_table.c file in the SDK does not use unlimited regulatory timers.

  • E910 SJC RxAtten of 3 dB may not converge to a correct solution (PI-23783)

    • The E910 SKU does not support 3dB Rx attenuation. For E910, the receiver attenuation selected via RxAtten field in the SjcGainControl register should not be set to Atten_3_dB. Setting the Rx attenuation on E910 to 3 dB may cause SJC convergence issues.

Planned Changes in Future Releases¶

Deprecation of Ex10Reader interface¶

The legacy ‘Ex10Reader’ interface, and support files, has been deprecated and will be removed in a future release. Impinj recommends updating applications to use the new Use case approach introduced in SDK version 2.0.

Deprecation of RF modes¶

Support for the following reader modes have been deprecated and will be removed in a future release. Impinj recommends updating applications to use the supported modes that have equivalent Gen2 parameters to deprecated modes.

Previous Releases¶

v1.2.0¶

Features added¶

  • Added new RF Mode 202 for faster read rates in ETSI lower band. Configured using RfMode register and SetRFModeOp.

  • Added Tx Power droop compensation:

    • Removed the PowerControlInitialTxScalar register to unify control in the TxFineGain register. Now the RadioPowerControlOp uses the same Tx scalar register used for the SetTxFineGainOp. This register is used as the starting value of the control loop and is also updated with the values obtained in running the loop.

    • Reader chip firmware will periodically adjust the TxFineGain register, so that power droop is corrected, staying closer to the target. This feature is enabled by default using register PowerDroopCompensation.

  • Added a new LBT (listen before talk) op ListenBeforeTalkOp.

  • Low power improvements:

    • Added SDK Power Modes API for removing power from the Impinj Reader Chip, placing the Impinj Reader Chip in low power mode as well as the power amplifier (PA).

    • In the SDK, when the TxRampUpOp is configured in the function Ex10Reader.build_cw_configs() the PA bias is adjusted based on the requested power level. The threshold for the PA bias adjustment is determined by the value LOW_BIAS_TX_POWER_MAX_CDBM found in the board_spec_constants.h file.

  • Ramp up improvements:

    • Added callback functions for pre- and post- ramp up functionality. These callbacks are separate, as are their register/unregister functions. They are executed in the ops layer cw_on function. The pre-ramp up callback happens before the ramp up in the function, and the post-ramp up callback happens after the ramp up in the function.

    • Added reverse power detector threshold detection. This allows us to detect when a larger power is read on the reverse power detector than anticipated and thus detect potentially dangerous situations for the hardware. The detection mechanism has been given a function call in the reader layer which can be called to detect this high Rx reading at any time. In addition, this call has been integrated into the post-ramp up callback.

  • Added a new interrupt mask to allow for interrupt on completion of the AggregateOp.

  • Added a new identifier to the AggregateOp FIFO summary along with a new command to update the identifier.

  • Added new GPIO pin accessor registers and SetClearGpioPinsOp which can be accessed by the added C SDK Ex10Ops.set_clear_gpio_pins() function.

  • Added BER functionality to the reader layer, thus making it simpler to use.

  • Added decoding of XPC words 1 and 2

  • Added a new HPF override test op. This op allows for tests to be performed which require overriding the internal high pass filter.

Fixes¶

  • Tx Power overshoot at ramp up due to tx_scalar default = 1152 (PI-27139).

    The Tx waveform overshoot at initial ramp up may be higher than the 5% specified by the Gen2 spec. The initial tx_scalar value may be lowered to help with this issue and the required infrastructure was put in place in calibration.c. By default, the drop percentage is set to 0%, leaving the tx_scalar unchanged. Refer to get_adjusted_tx_scalar() for details and to make adjustments to the default drop percentage.

  • FastID feature would misclassify an EPC as an empty EPC with a TID attached (PI-27567).

  • EBV calculation errors (PI-27282).

    There is an initial length calculation in the Gen2 encode function which is only used for setting the encode array to 0. The actual encoding itself was fine, but memory at the end of the encoded function was not cleared properly. Since the encode is an OR’ing process, if the backing memory was not previously zeroed, the encoded command ran the chance of being incorrect.

  • Channel off-times may not be observed (PI-28439).

    Off time on the same channel was not being observed/used even though it was defined in the region table of the SDK. The off time has been removed, and off_time_same_channel is now being used in the CW on process to ensure we are not ramping too soon if ramping on the same channel we just ramped down on.

  • SDK make with BUILD_BASE does not work (PI-28284)

    Now, specifying make BUILD_BASE=<build_path> will output the build artifacts into the directory path specified by BUILD_BASE.

  • Double-start of inventory from SDK (PI-27466).

    If ops inventory() was called while the inventory op was already running, it would start an aggregate op to set the RF mode that would be ignored by the Ex10 device and then would wait for op completion which would actually be the inventory op completing. To protect against this issue, a check was added to see if Ex10 device is idle before attempting to start an inventory round from the SDK.

  • Certain tags cannot be inventoried with FastId enabled (PI-29082).

    FastId operation was not waiting for the T4 time between the Select and the Query commands. This would result in some tags not seeing the Query command as the preceding Select would violate T4 time, causing these tags to not be properly inventoried when FastId is enabled.

  • PC calibration script in v1.1.0 does not properly run (PI-27639, PI-27641).

    v1.1.0 PC calibration script does not properly run due to incorrect import statements and missing default parameter for serial port.

  • Erroneous slot classification in some RF Modes (YUKON-6430).

    Some RF modes misclassified collided slots as empty, causing early termination of the inventory round. Built-in thresholds were adjusted to significantly improve slot classification.

  • Occasional hangs during continuous inventory (PI-28188).

    Occasionally, the inventory operation would hang when the struct ContByteSpan bytes used in ex10_reader.c fifo_data_handler() would be incorrectly set to zero. This would result in missing the Inventory Summary packet and the restart of inventory would not be performed. To fix, reader_list_node_push_back() in fifo_data_handler was moved to after continuous inventory state processing has completed.

  • Large Tx power error when target is out of range (PI-28457)

    Added checks to verify that target power is within calibrated range and added caps when requested power is outside the calibrated range.

  • pc_cal_test.py imports with “cal.” fail (PI-29865)

    When a bundle is extracted, pc_cal_test.py is in a directory “ex10_cal” but imports of some python modules assume a directory name of “cal”. Removing “cal.” prefix allows pc_cal_test.py to import the modules and run normally.

Known issues¶

  • Using an unlimited regulatory timer may cause poor reader performance (PI-29431)

    The Impinj Reader Chip SDK performs certain reader chip configuration steps only when the reader is ramped down and ramped back up. For example, temperature measurement for LO PDET ADC target selection and PLL configuration. If the reader temperature changes drastically without the reader ramping down and back up, the Tx power will become inaccurate, and the PLL could become unlocked. Impinj recommends partners do not use an unlimited regulatory timer for this reason. Impinj’s implementation of the regions_table.c file in the SDK does not use unlimited regulatory timers.

  • E910 SJC RxAtten of 3 dB may not converge to a correct solution (PI-23783)

    The E910 SKU does not support 3dB Rx attenuation. For E910, the receiver attenuation selected via RxAtten field in the SjcGainControl register should not be set to Atten_3_dB. Setting the Rx attenuation on E910 to 3 dB may cause SJC convergence issues.

v1.1.0¶

Features added¶

  • Added new reader modes. For detailed updated list of modes see SetRFModeOp.

  • Improved Tx waveforms for FCC DRM and ETSI low-band DRM reader modes.

  • Auto-Access - ability to execute a pre-configured sequence of Gen2 commands on every singulated tag without additional signaling from the host.

    Controlled from InventoryRoundControl and Gen2AutoAccessEnable

  • Q/Target behavior for continuous inventory updated, details in Inventory Round Q/Target Behavior for Continuous Inventory

  • Aggregate Op - allows commanding Ex10 IC to execute a user-defined sequence of Ops and commands, optimizing host side control by reducing SPI traffic between Ex10 IC and the host controller.

    For details, see AggregateOp.

    In addition, helpers were added to the host SDK for building the user-defined sequence.

  • RSSI filtering - filtering tag responses based on user-adjustable RSSI thresholds for LMAC RN16 and EPC, controlled respectively from RssiThresholdRn16 and RssiThresholdEpc

  • IRQ_N may be configured to trigger on unsuccessful completion of a response-less command.

    Controlled from InterruptMask register.

  • Added UsTimerStartOp and UsTimerWaitOp.

  • Ex10 wrapper module expanded to support the following:

    • Set GPO

    • Application image upload

    • Read firmware version information

    • Read info pages

    • Start PRBS

    • Issue Reset command

    • Start inventory

  • Gen2 SDK module refactored for ease-of-use and pliability, for details see Gen2 Tx Command Manager Usage.

  • Inventory and Continuous-inventory examples updated to use dynamic Q for better read rates.

  • PowerControlLoopOp was updated to rely on 2 thresholds - LoopStopThreshold used as a stop condition for the internal control loop and OpErrorThreshold used to determine Success/Error status returned by the Op in OpsStatus register.

    For details see PowerControlLoopAdcThresholds.

    Note that the previous register name ‘PowerControlLoopAdcErrorThreshold’ was changed to PowerControlLoopAdcThresholds and previous internal field ‘ErrorThreshold’ renamed to ‘LoopStopThreshold’.

  • Add register SjcResidueThreshold which defines the pass/fail threshold when running the RxRunSjcOp.

  • The RxRunSjcOp may now fail due to CDAC range error or the residue threshold error.

  • Removed constant setting of Rx gains in EtsiBurstOp to improve channel hop time

  • Firmware upload APIs exposed in SDK protocol layer for start, continue and complete of an application image upload.

  • Added Halted Event FIFO packet

  • Larger passing margins for adjacent channel selectivity, which is part of the ETSI regulatory requirement on EU2.

Fixes¶

  • SJC algorithm updated to consistently find the correct solution for low BLF values (PI-24355)

    For higher self-jammer levels, SJC would sometimes fail to find the optimal solution. The failures were often but not always detectable (residue magnitude > 10,000).

  • Initial Tx coarse gain of 0 was not properly passed to the Ex10 IC (PI-25743)

    In v1.0, the SDK would cache Op parameters to minimize SPI traffic. When trying to set the initial Tx coarse attenuation of 0, the SDK would not configure TxCoarseGain register and the Ex10 internal register would remain set to higher default value, resulting in incorrect Tx power. This bug was fixed by removing the caching behavior from the SDK (in this case, removing stored_coarse_gain).

  • Continuous inventory in the reader layer of the C-SDK was capable of hanging. This was due to improper handling of invalid packet data from the device event fifo buffer. When the data was received, the data fifo handler would attempt to parse the invalid data and could see it as good with an incorrect length. This parsing of garbage lead us into a repeating loop (PI-26518).

  • Now using mutex locking for independent locking of the free list(which takes in data from the device event fifo buffer) and the reader list (pending events which are to be parsed by the user at the reader layer) (PI-26518).

  • The Tx waveform overshoot is higher than that specified by the Gen2 spec. To combat this, the function get_adjusted_tx_scalar in calibration.c can be used to drop the tx_scalar value and thus the output by a user specified percentage. This percentage is hard-coded in the function and should be changed based on customer usage. The default of this function is to drop the output by 0%.

Known issues¶

  • The Tx waveform overshoot is higher than that specified by the Gen2 spec. To combat this, the tx_scalar value should be lowered in the future.

  • Low power idle mode not implemented (PI-25286).

  • AggregateOp can not use an InsertFifoEvent command with a NULL packet. Normally, the InsertFifoEvent command can take the command code and trigger interrupt with no packet payload provided. The aggregate op helper and aggregate op in general should not be used this way with this version. If utilizing the aggregate op, use a non-zero length packet payload with it (PI-25971).

  • When using the reader level continuous inventory functionality, rollover timing will not control the inventory stop conditions correctly. The timing is a uint32_t in us, and if it rolls over, the math for time elapsed will be inaccurate. This timing should not be relied on for accuracy. We recommend either starting a new continuous inventory if you are not impacted by the inaccuracy or using a different timing mechanism.

v1.0.250-rc (1.1 release candidate)¶

A release candidate to v1.1.0 release, included all new features and fixes except for the two changes implemented to fix PI-26518.

v1.0.12 (1.0 patch)¶

Patch to v1.0.0 release, replaced v1.0.0 as the initial production release.

Features added¶

  • Support added for newer Raspberry Pi Model 4 version (RPi BCM2711 GPIO label added)

Fixes¶

  • ReqRN response and CRC (PI-24582): When the reader chip receives a ReqRN response with a bad CRC, the lower-level radio control (LMAC) will hang and tag reading will stop until the LMAC restarts operation (ramps down on current channel and ramps back up on the next channel). LMAC state machine was fixed as follows:

    • If LMAC receives a bad response for a ReqRn that was issued to move a tag to open state, LMAC will move to the next slot.

    • If LMAC receives a bad response to a ReqRn that was issued to retrieve a cover code from a tag, LMAC will report failed CRC via ‘status’ field in Gen2 transaction Event FIFO packet and return to Halted state to wait further instructions from the host

  • EPC and TID corruption (PI-24609): When the reader chip is operated at high tag read rates, EPCs and TIDs reported to the host could be corrupted, and in some cases, the firmware could become unresponsive. If too many internal LMAC events occurred in a short amount of time, the device would crash. The LMAC internal behavior was changed to stop and report via the stop reason in Inventory round summary Event FIFO packet.

  • Documentation error regarding reader modes (PI-25285): SDK documentation previously implied that reader modes can be changed using SetRFModeOp while the CW is on. Documentation has been updated to clarify that the CW must be ramped down before changing reader modes.
